# HTTP flow

This is the practical request sequence for host code that interacts with Codex Broker.

Set examples to match your deployment:

```bash
BROKER=http://127.0.0.1:3400
BROKER_KEY=dev-only-key
OWNER=service-account-1
```

If an `ownerId`, `threadId`, or `turnId` contains `/`, spaces, or other reserved characters, URL-encode that path segment. The bundled Python and TypeScript clients do this for you.

These examples use the default Codex auth profile. Select the profile when creating the broker thread. A turn cannot override the thread's auth principal or profile.

`ownerId` scopes broker state and audits. Trusted policy resolves the upstream `authPrincipalId`, defaulting to the owner. A request may assert the configured value but cannot select an arbitrary account.

## Check auth status

```bash
curl -sS \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/auth/status"
```

`authFingerprint` changes when the auth-principal/profile auth file changes. App-server children are keyed by that fingerprint so refreshed auth starts fresh runtime processes.

List last-recorded profile state without invoking Codex:

```bash
curl -sS \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/auth/profiles"
```

## Run an active probe

Use a probe for manual checks or low-frequency health workflows, not frequent polling:

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{}' \
  "$BROKER/v1/owners/$OWNER/auth/probe"
```

If Codex reports token invalidation, the broker stores `refresh_failed` and closes stale pooled app-server children for that auth-principal/profile pair.

## Check usage and rate limits

These reads query the selected auth-principal/profile without starting a turn:

```bash
curl -sS \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/auth/usage"

curl -sS \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/auth/rate-limits"
```

The evolving Codex payload is returned under `usage` or `rateLimits`. `authPrincipalHash` identifies the upstream total; mapped owners share it while `ownerHash` and audits remain separate.

## Consume a rate-limit reset credit

This changes account state. Only call it after an explicit user or administrator action, and reuse the same idempotency key when retrying that action:

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{"idempotencyKey":"incident-123-reset"}' \
  "$BROKER/v1/owners/$OWNER/auth/rate-limit-reset-credit/consume"
```

The broker records successful consumption in the owner audit log.

## Store an API key

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{"apiKey":"'"$OPENAI_API_KEY"'"}' \
  "$BROKER/v1/owners/$OWNER/auth/api-key"
```

## Invalidate runtime auth

If an administrator refreshes shared auth outside the broker, close that Codex auth profile's pooled app-server children before retrying failed work:

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{}' \
  "$BROKER/v1/owners/$OWNER/auth/runtime/invalidate"
```

To replace the account inside a profile, quiesce all sharing owners, logout with `deleteProfile: true`, authenticate the replacement, and use a new broker thread id. Profile deletion changes the binding instance, so old and queued threads fail closed.

## Create or reuse a thread

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "threadId": "chat-123",
    "profile": "default",
    "hostApp": "chat-app",
    "bundleId": "example-chat-v1",
    "configProfile": "default",
    "cwd": "/workspaces/app"
  }' \
  "$BROKER/v1/owners/$OWNER/threads"
```

Response:

```json
{
  "threadId": "chat-123",
  "codexThreadId": null,
  "authPrincipalHash": "...",
  "profile": "default",
  "configProfile": "default",
  "hostApp": "chat-app",
  "bundleId": "example-chat-v1",
  "cwd": "/workspaces/app",
  "status": "active",
  "createdAt": "2026-06-28T12:00:00Z",
  "updatedAt": "2026-06-28T12:00:00Z"
}
```

## Submit a turn

```bash
THREAD_ID=chat-123

curl -sS \
  -X POST \
  -H "Authorization: Bearer $BROKER_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "input": [
      { "type": "text", "text": "Summarize the evidence for this user question." }
    ],
    "hostApp": "chat-app",
    "bundleId": "example-chat-v1",
    "configProfile": "default",
    "codexOptions": {
      "model": "gpt-5.6-sol",
      "effort": "high"
    },
    "mode": "queue",
    "productCorrelationId": "chat-123:message-456",
    "idempotencyKey": "chat-123:message-456"
  }' \
  "$BROKER/v1/owners/$OWNER/threads/$THREAD_ID/turns"
```

Each turn can choose a model with `codexOptions.model` and reasoning effort with `codexOptions.effort`. Request values override the selected `configProfile`; profile values override Codex's own defaults. If neither layer sets a value, the broker lets Codex choose its current recommended model and model-specific reasoning default. The `reasoningEffort` alias is also accepted, and supported effort values depend on the selected model.

Response:

```json
{
  "threadId": "chat-123",
  "turnId": "turn_...",
  "codexTurnId": null,
  "authPrincipalHash": "...",
  "profile": "default",
  "configProfile": "default",
  "hostApp": "chat-app",
  "bundleId": "example-chat-v1",
  "cwd": "/workspaces/app",
  "mode": "queue",
  "productCorrelationId": "chat-123:message-456",
  "status": "queued",
  "error": null,
  "errorCode": null,
  "publicMessage": null,
  "adminMessage": null,
  "createdAt": "2026-06-28T12:00:01Z",
  "startedAt": null,
  "completedAt": null,
  "updatedAt": "2026-06-28T12:00:01Z",
  "streamUrl": "/v1/owners/service-account-1/threads/chat-123/events?turnId=turn_..."
}
```

`streamUrl` is relative to the broker base URL and already filters to the submitted turn. It does not include the auth header.

## Stream events

```bash
TURN_ID=turn_...

curl -N \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/threads/$THREAD_ID/events?turnId=$TURN_ID&after=0"
```

Use `after=<last-event-id>` to resume without replaying already processed events.

## Poll the turn

```bash
curl -sS \
  -H "Authorization: Bearer $BROKER_KEY" \
  "$BROKER/v1/owners/$OWNER/threads/$THREAD_ID/turns/$TURN_ID"
```

Terminal statuses are `completed`, `failed`, `timed_out`, and `interrupted`. Nonterminal statuses are `starting`, `queued`, and `running`.

## Error handling

Broker HTTP errors use JSON:

```json
{ "error": "Thread not found." }
```

Common statuses:

| Status | Meaning |
| --- | --- |
| 400 | Invalid request body, invalid mode, invalid bundle, invalid `cwd`, or policy validation failure. |
| 401 | Missing or invalid broker key. |
| 404 | Unknown route, thread, turn, or stream filter target. |
| 409 | Active turn conflict, archived thread, shutdown, or other conflict. |
| 502 | Codex app-server request failed. |
| 500 | Unexpected broker error. |
